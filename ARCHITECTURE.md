# Architecture

현재는 제품 코드가 없으므로 이 문서는 목표 아키텍처와 제약을 정의한다.

## Current State

- 리포지터리는 초기 부트스트랩 상태다.
- 제품 요구사항은 초기 기획서 기준으로 1차 확인됐다.
- 제품은 앱 우선, 추후 웹 확장을 염두에 둔 TO DO/생활계획표 서비스다.
- 하네스부터 먼저 세운 뒤 기능 구현을 시작한다.

## Target Layout

## Product Direction From Planning Doc

- 프로젝트명: `오늘 다 했니?`
- 제품 유형: TO DO 웹앱 성격의 서비스
- 기획 배경: 계획만이 아니라 실천까지 관리
- 기획 목적: 습관 형성을 위한 세밀한 계획과 실천 시스템 구축
- 타깃 사용자: 일정을 계획하고 실천하는 사람들
- 주요 기능: 생활계획표 등록, 알람, 완료 여부 관리
- 플랫폼 방향: 앱 우선, 추후 웹 확장

## Architectural Principles

- 비즈니스 로직은 도메인 중심으로 분리한다.
- 외부 시스템 연동은 `providers/` 경계로 수렴시킨다.
- UI는 도메인 로직을 직접 가지지 않는다.
- 스키마 검증은 경계에서 수행한다.
- 로깅은 구조화된 이벤트 단위로 남긴다.
- 웹 MVP 이후 앱 전환을 고려해, 시간 계산과 일정 규칙은 플랫폼 독립 모듈로 유지한다.
- 알림, 저장소, 시간 소스는 provider 인터페이스 뒤로 숨긴다.

## Intended Layering

각 도메인은 아래 순서를 따른다.

1. `types`
2. `config`
3. `repo`
4. `service`
5. `runtime`
6. `ui`

허용 원칙:
- 상위 레이어는 바로 아래 레이어 또는 명시적 provider만 사용한다.
- 교차 관심사는 provider 인터페이스를 통해 주입한다.
- 임시 유틸리티보다 재사용 가능한 공유 패키지/모듈을 우선한다.

## Enforcement Candidates

- 폴더 구조 검증
- 파일 크기 제한
- 아키텍처 import 규칙
- 문서 존재 여부 검증
- 주요 계획 문서 링크 검증

## Web To App Transition Constraint

- 웹 MVP 단계에서 검증할 대상은 문제-해결 적합성과 핵심 사용자 흐름이다.
- 앱 전환 시 재사용해야 하는 것은 도메인 모델, 시간 계산, 상태 규칙, 저장소 인터페이스, 알림 인터페이스다.
- 웹 전용 렌더링 구현은 `ui/` 경계에 가둔다.
