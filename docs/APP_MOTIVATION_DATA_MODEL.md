# App Motivation Page And Data Model

## Goal

월간 동기부여 페이지를 현재 버전에 포함하기 위해 필요한 데이터 모델과 저장 방향을 고정한다.

이 문서는 `오늘 화면` 중심 제품을 유지하면서도, 사용자가 `얼마나 지켰는지`, `몇 개를 완료했는지`, `연속으로 얼마나 해냈는지`를 볼 수 있게 만드는 최소 구조를 정리한다.

## Product Decision

- 월간 동기부여 페이지는 `나중 기능`이 아니라 현재 버전 범위에 포함한다.
- 따라서 저장 모델은 더 이상 `오늘 계획 1묶음`만 전제로 두지 않는다.
- 첫 앱 저장소는 여전히 `AsyncStorage` 계열로 시작할 수 있지만, 데이터 모델은 날짜 기반 누적 구조로 설계한다.
- 월간 지표가 핵심 화면으로 유지되면 중기적으로 `SQLite` 전환 가능성을 기본 전제로 둔다.

## Motivation Page Goal

사용자가 앱을 계속 쓰게 만드는 최소 동기부여 신호를 제공한다.

첫 버전에서 우선 보여줄 값:

- 이번 달 완료한 계획 개수
- 이번 달 완료율 평균 또는 누적 완료율
- `missed` 개수
- 재지정 후 결국 완료한 개수
- 연속 달성 일수

보여주지 않을 것:

- 과한 랭킹/경쟁 요소
- 감정 분석
- 복잡한 생산성 점수
- 장문 리포트

## New Core Requirement

월간 동기부여 페이지를 넣으려면 `오늘 상태`와 `기록`을 분리해야 한다.

필요한 구분:

- 오늘 편집 중인 계획 상태
- 날짜별로 확정된 하루 기록
- 월간 집계를 위한 요약 기준

즉, 저장은 최소한 아래 세 층을 염두에 둬야 한다.

1. 특정 날짜의 계획 목록
2. 각 계획의 완료/놓침/재지정/회고 정보
3. 날짜 단위 요약 또는 날짜 기준 집계 가능 구조

## Date-Based Model Draft

### A. Daily Plan Record

하루 계획은 최소한 `date`를 가져야 한다.

예시 필드:

- `id`
- `date`
- `title`
- `startMinute`
- `endMinute`
- `status`
- `color`
- `rescheduleCount`
- `sourcePlanId`
- `reflectionNote`

핵심 원칙:

- `date`가 빠진 계획 모델은 월간 집계 기준으로 쓸 수 없다.
- `sourcePlanId`는 같은 일정 체인의 재지정 흐름을 월간 통계에서 다시 묶는 데 필요하다.

### B. Daily Summary Snapshot

날짜별 요약은 매번 전부 다시 계산할 수도 있지만, 최소 구조는 먼저 정의해 둔다.

예시 필드:

- `date`
- `plannedCount`
- `completedCount`
- `missedCount`
- `completionRate`
- `rescheduledCount`
- `completedAfterRescheduleCount`
- `reflectionCount`

첫 버전 원칙:

- 처음부터 별도 영속 테이블로 두지 않아도 된다.
- 하지만 이 값들을 계산할 수 있는 원본 필드는 반드시 저장돼야 한다.

### C. Observation Data

관찰 로그는 장기 제품 데이터의 중심은 아니지만, 월간 동기부여와 완전히 분리된 저장으로 본다.

이유:

- 리마인드 반응 로그는 제품 정책 점검용이다.
- 사용자의 월간 성취 집계와 같은 우선순위로 섞어 저장할 필요는 없다.

## First Motivation Metrics

초기 월간 페이지는 아래 지표면 충분하다.

1. `이번 달 완료 개수`
2. `이번 달 완료율`
3. `이번 달 missed 개수`
4. `재지정 후 완료 개수`
5. `연속 완료 일수`

계산 기준 초안:

- 완료 개수: 해당 월의 `done` 상태 계획 수
- 완료율: `done / 전체 계획`
- missed 개수: 해당 월의 `missed` 상태 계획 수
- 재지정 후 완료 개수: `sourcePlanId`가 있는 계획 중 `done` 상태 수
- 연속 완료 일수: 하루 완료율 기준 임계값을 넘긴 날의 연속 개수

## Storage Direction

### Start Now

- 첫 앱 저장소는 `AsyncStorage` 계열로 시작 가능
- 단, 저장 키 내부 구조는 날짜 기반 문서 또는 날짜별 레코드 집합을 담을 수 있어야 함

### Prepare For Later

- 월간 페이지가 실제 사용 흐름에 포함되면, 조회와 집계가 반복된다
- 이 시점부터는 `SQLite`가 더 자연스러운 구조가 된다

실무 판단:

- 현재 버전에 월간 동기부여 페이지를 넣는다면 `SQLite` 전환을 전혀 고려하지 않는 설계는 피해야 한다
- 그러나 앱 첫 구현을 너무 무겁게 시작하지 않기 위해 저장소 자체는 바로 `AsyncStorage`로 시작할 수 있다

## Transition-Safe Rules

저장소 전환 비용을 낮추기 위해 아래 규칙을 지킨다.

1. 계획 모델에 `date`를 반드시 둔다.
2. 재지정 체인을 위해 `sourcePlanId`를 유지한다.
3. 화면은 월간 값을 직접 저장소에서 조립하지 말고, 집계 함수나 selector 계층을 거친다.
4. 관찰 로그와 제품 기록을 분리한다.
5. 저장소 구현 세부사항은 계속 provider 뒤에 둔다.

## Recommended Rollout

1. 날짜 기반 데이터 모델 초안을 먼저 고정
2. `PlansStore` 계약이 날짜 기반 구조를 감당할 수 있게 다음 개정 방향 검토
3. 초기 앱 저장소는 `AsyncStorage`
4. 월간 페이지 시안과 지표 정의 동시 진행
5. 월간 집계가 핵심 흐름으로 굳으면 `SQLite` 전환 시점 확정

## Follow-up

이 문서 다음으로 필요한 것은 아래다.

1. 저장소 전환 대비 규칙을 `APP_STORAGE_OPTIONS.md`와 맞물리게 보강
2. 월간 동기부여 페이지 화면 초안 추가
3. 앱 로컬 알림 구현 초안 작성
