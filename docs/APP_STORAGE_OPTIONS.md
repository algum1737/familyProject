# App Storage Options

## Goal

앱 전환 시 `PlansStore` 계약 뒤에 붙일 네이티브 저장소 후보를 비교하고, 첫 구현 권고안을 고정한다.

이 문서는 초기 앱 범위에서 무엇이 충분하고, 무엇이 과한지 판단하기 위한 선택 문서다.

## Current Contract

현재 웹 구현의 저장소 계약은 아래 두 동작만 가진다.

- `load(): DailyPlan[] | null`
- `save(plans: DailyPlan[]): void`

즉, 지금 단계의 기본 저장 요구는 아래 두 층으로 본다.

- 오늘 화면을 위한 계획 읽기/저장
- 월간 동기부여 페이지를 위한 날짜 기반 누적 기록 유지

아직 없는 요구:

- 다중 사용자 동기화
- 서버 병합
- 복잡한 관계형 질의
- 대량 데이터 pagination

## Decision Criteria

후보 비교 기준은 아래 다섯 가지로 고정한다.

1. 현재 `PlansStore` 계약에 무리 없이 붙일 수 있는지
2. 초기 앱 범위에 비해 저장소가 과하지 않은지
3. 오프라인 단일 사용자 저장에 충분히 안정적인지
4. 이후 라벨 설정, 관찰 로그 같은 소규모 추가 저장도 같이 감당하기 쉬운지
5. 나중에 더 강한 저장소로 옮길 때 교체 비용이 낮은지

## Candidate A: AsyncStorage

설명:

- React Native/Expo 계열에서 가장 단순하게 붙일 수 있는 key-value 저장소다.

장점:

- 현재 `load/save` 계약과 가장 직접적으로 맞는다.
- 계획 배열 전체를 JSON으로 저장하는 현재 웹 구조를 거의 그대로 옮길 수 있다.
- 구현 비용이 가장 낮다.
- 라벨 설정, 관찰 로그 같은 작은 설정성 데이터도 같은 계층으로 같이 저장할 수 있다.

단점:

- 부분 업데이트나 복잡한 조회가 비효율적이다.
- 데이터가 커지면 전체 JSON 읽기/쓰기 비용이 커진다.
- 관계형 질의나 복잡한 오프라인 모델로 확장할 때 한계가 빨리 온다.

적합도:

- 현재 프로젝트의 첫 앱 버전에는 충분히 적합하다.

## Candidate B: MMKV

설명:

- 빠른 key-value 저장소 계열이다.

장점:

- 단순 키-값 저장에서 성능이 좋다.
- 설정값, 라벨, 작은 문서 저장에는 잘 맞는다.
- AsyncStorage보다 반응성이 좋을 수 있다.

단점:

- 현재 프로젝트의 저장량 기준으로 성능 이점이 제품 가치로 바로 이어질 가능성은 낮다.
- Expo 관리 워크플로우와 실제 적용 방식까지 같이 고려해야 한다.
- 첫 버전에서 필요한 것은 성능 최적화보다 저장 흐름의 단순함이다.

적합도:

- 가능은 하지만, 첫 앱 버전의 기본 선택으로는 과할 수 있다.

## Candidate C: SQLite

설명:

- 기기 내 관계형 데이터베이스다.

장점:

- 일정이 날짜 단위로 크게 늘어나거나, 검색/필터/통계가 붙을 때 확장성이 높다.
- 부분 조회와 구조화된 저장에 유리하다.
- 이후 히스토리, 주간/월간 조회, 분석 기능 확장에 मजबूत한 기반이 된다.

단점:

- 현재 `load/save` 계약보다 훨씬 큰 저장 모델을 먼저 요구한다.
- 스키마, 마이그레이션, 레코드 단위 설계를 함께 열어야 한다.
- 첫 앱 버전의 단일 사용자 하루 플래너 범위에는 분명히 과하다.

적합도:

- 중장기 확장 후보로는 좋지만, 지금 바로 기본 선택으로 쓰기엔 무겁다.

## Candidate D: Realm or Similar Object Database

설명:

- 객체 모델 중심 로컬 DB 계열이다.

장점:

- 복잡한 로컬 데이터 모델과 반응형 갱신에 강하다.
- 향후 확장 범위가 매우 넓다.

단점:

- 현재 범위 대비 도입 비용이 과하다.
- 저장 모델과 라이브러리 결합이 강해질 수 있다.
- 지금 필요한 것은 단순한 앱 저장소 provider이지, 전체 로컬 데이터 플랫폼이 아니다.

적합도:

- 현 단계에서는 부적합하다.

## Comparison Summary

### Best Fit Now

- `AsyncStorage`

이유:

- 앱 첫 구현을 무겁게 시작하지 않으면서도 날짜 기반 문서 저장으로 출발할 수 있기 때문이다.
- 웹 `localStorage` 구현을 앱으로 옮길 때 개념 차이가 가장 작다.
- 첫 앱 목표가 저장 기술 검증이 아니라 오늘 화면 실행 흐름과 월간 동기부여 흐름을 함께 복구하는 일이기 때문이다.

### Best Upgrade Path Later

- `SQLite`

이유:

- 날짜 누적, 히스토리, 통계, 검색, 동기화 전 단계 구조로 확장될 때 가장 자연스럽다.

### Not Recommended As Default First Choice

- `MMKV`
- `Realm` 계열

이유:

- 현재 범위에서 제품 리스크보다 저장소 기술 선택 비용이 더 커질 가능성이 높다.

## Recommendation

초기 앱 저장소 기본안은 `AsyncStorage` 계열로 간다.

구체 원칙:

- `PlansStore` 계약은 그대로 유지한다.
- 앱 구현에서는 `localStorage` 대신 네이티브 key-value 저장으로 교체한다.
- 계획 데이터는 날짜 기반 문서 또는 날짜별 레코드 집합을 JSON으로 저장한다.
- 라벨 설정과 관찰 로그도 같은 수준의 key-value 저장으로 유지한다.

이 선택이 맞는 이유:

- 현재 제품 범위는 단일 사용자, 단일 기기 중심이지만, 월간 동기부여 페이지를 위해 날짜 기반 누적 기록이 필요하다.
- 저장소보다 중요한 것은 현재 계획 인지, 완료, 리마인드, `missed` 회복, 월간 동기부여 흐름을 먼저 앱에서 되살리는 일이다.
- 더 무거운 DB는 월간 조회와 집계가 실제 핵심 흐름으로 굳어질 때 도입해도 늦지 않다.

## Migration Trigger

아래 조건 중 둘 이상이 생기면 `SQLite` 계열로 재검토한다.

- 날짜를 넘는 다일자 히스토리를 기본 기능으로 넣어야 한다.
- 계획 배열 전체 덮어쓰기보다 부분 수정 빈도가 훨씬 높아진다.
- 통계/검색/필터 기능이 제품 핵심으로 올라온다.
- 로컬 저장 데이터 양이 체감될 정도로 커진다.
- 서버 동기화 이전에 구조화된 로컬 레코드 모델이 필요해진다.

## App Provider Shape

앱 첫 구현에서는 아래 형태면 충분하다.

- `load()`에서 저장 문자열을 읽고 JSON 파싱 후 `validatePlanner`로 검증
- `save()`에서 계획 배열 전체를 직렬화해 저장
- 저장 키는 웹과 논리적으로 같은 도메인 이름을 유지

단, 월간 동기부여 페이지를 현재 버전에 포함하므로 실제 저장 값은 `오늘 배열 1개`보다 날짜 기반 누적 구조를 담을 수 있어야 한다.

예시 방향:

- plans: `today-did-you-finish:plans`
- labels: `today-did-you-finish:labels`
- reminder observations: `today-did-you-finish:reminder-observations`
- recovery observations: `today-did-you-finish:recovery-observations`
