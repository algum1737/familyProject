# Web To App Transition

## Goal

웹 MVP에서 핵심 가치와 사용자 흐름을 검증한 뒤, 실제 앱 전환을 실행하기 위한 기술적 경계와 전환 조건을 미리 정의한다.

## Strategy

1. 웹 MVP로 문제-해결 적합성을 먼저 검증한다.
2. 도메인 로직, 시간 계산, 일정 규칙은 플랫폼 독립 모듈로 유지한다.
3. 저장소, 알림, 시간 소스는 provider 인터페이스로 추상화한다.
4. 앱 전환 시 UI 레이어를 우선 교체하고, 도메인과 규칙은 최대한 재사용한다.

## Why Web First

- 원형 24시간 계획판과 현재 시각 포인터 UX를 빠르게 검증할 수 있다.
- 시간 입력 방식의 사용성 문제를 저비용으로 수정할 수 있다.
- 핵심 여정 테스트와 시각 검증을 빠르게 반복할 수 있다.
- 앱 전환 전 제품 방향 오류를 줄일 수 있다.

## Reusable Boundaries

앱 전환 시 재사용 대상:

- 일정 데이터 모델
- 시간 슬롯 계산 로직
- 현재 시각이 속한 계획 판정 로직
- 완료 상태 규칙
- 요약 계산 로직
- 저장소 인터페이스
- 알림 인터페이스

앱 전환 시 재작업 가능성이 높은 영역:

- 원형 계획판 렌더링 구현
- 입력 인터랙션
- 네이티브 알림 연동
- 플랫폼별 네비게이션과 화면 전환

## Transition Trigger

아래 조건이 충족되면 앱 전환 검토가 아니라 앱 전환 실행 작업에 들어간다.

1. 핵심 사용자 여정이 테스트로 고정되어 있다.
2. 사용자가 현재 시각 기준으로 해야 할 계획을 이해한다는 정성 피드백이 반복 확인됐다.
3. 일정 등록부터 완료 처리까지의 흐름에서 주요 UX 혼란이 해결됐다.
4. 웹 MVP 사용 데이터 또는 테스트 기록이 남아 있다.
5. 도메인 로직이 UI 코드에서 분리되어 있다.

## Validation Signals

정성 신호:

- 사용자가 첫 화면을 보고 현재 해야 할 계획을 바로 설명할 수 있다.
- 사용자가 시간 입력 방식을 부담 없이 받아들인다.
- 원형 계획판이 목록형 UI보다 더 직관적이라는 피드백이 나온다.

정량 신호:

- 첫 세션에서 일정 등록 성공
- 현재 계획 확인 후 완료 처리까지 이어지는 세션 존재
- 핵심 흐름 smoke test 안정 통과

## Transition Options

### Option A

`Expo + React Native`로 앱 UI를 다시 구성한다.

- 장점: 진짜 앱 경험, 로컬 알림과 디바이스 기능 확장에 유리
- 단점: UI 레이어 재구현 비용이 크다

### Option B

웹 앱을 하이브리드 앱 셸로 감싼다.

- 장점: 초기 코드 재사용 폭이 크다
- 단점: 복잡한 인터랙션과 네이티브 UX에서 제약이 있다

## Decision Criteria

앱 전환 방식은 아래 기준으로 비교한다.

1. 원형 24시간 플래너 인터랙션을 모바일에서 자연스럽게 구현할 수 있는지
2. 로컬 알림, 시간 기반 리마인드, 백그라운드 제약 대응이 충분한지
3. 현재 도메인 로직과 provider 경계를 얼마나 그대로 재사용할 수 있는지
4. 초기 앱 출시까지 필요한 UI 재구현 비용이 어느 정도인지
5. 이후 저장, 계정, 동기화 요구가 붙었을 때 확장성이 있는지
6. 앱 스토어 배포와 운영 루프를 무리 없이 붙일 수 있는지

## Decision Inputs

각 옵션 비교 전에 아래 입력을 먼저 정리한다.

- 초기 앱에서 반드시 제공해야 할 화면과 흐름
- 웹 MVP에서 사용자 검증이 더 필요한 UX
- 알림 기능의 최소 요구 수준
- 오프라인 사용과 저장 전략의 최소 요구 수준
- 앱 전환 시점까지 분리돼 있어야 하는 도메인/provider 경계

## Current Codebase Read

현재 코드 기준 판단:

- `src/domains/plans/service/planner.ts`의 시간 계산, 겹침 검증, 현재 계획 판정, 요약 계산은 플랫폼 독립 재사용 후보다.
- `src/providers/time/time-source.ts`는 인터페이스가 얇고, 현재는 `PlannerShell`을 통해 UI에 주입된다.
- `src/providers/plans/plans-store.ts`와 `src/providers/plans/local-plans.ts`로 저장소 계약과 웹 구현이 분리돼 있다.
- `src/ui/planner/circular-planner.tsx`는 `window`, `HTMLInputElement`, 브라우저 폼과 SVG 렌더링에 강하게 묶여 있어 UI 레이어 재구성이 거의 불가피하다.

요약하면, 도메인/규칙은 재사용성이 높고 현재 UI는 웹 전용 결합이 강하다.

## App UI Reuse Strategy

앱 전환 시 UI를 통째로 재사용하는 것이 아니라, 재사용 가능한 층과 재구성해야 하는 층을 분리해서 본다.

재사용 우선 대상:

- `src/domains/plans/service/planner.ts`의 시간 계산, 겹침 검증, 현재 계획 판정, 요약 계산
- `src/providers/plans/plans-store.ts`의 저장소 계약
- `src/providers/time/time-source.ts`의 시간 소스 계약
- `src/providers/reminders/reminder-provider.ts`의 리마인드 계약
- `src/ui/planner/use-planner-state.ts`의 플래너 편집 상태 전이와 저장 호출 흐름

앱에서 재구성할 대상:

- `src/ui/planner/circular-planner.tsx`의 SVG 기반 원형 플래너 렌더링
- 브라우저 폼과 `HTMLInputElement`를 전제로 한 입력 상호작용
- 웹 레이아웃과 CSS 클래스 기준의 시각 구성

권장 계층 구조:

1. 도메인과 provider 계약은 웹/앱 공통 모듈로 유지
2. 플래너 상태 전이 hook은 웹 구현 세부사항이 더 스며들지 않게 유지
3. 웹 `CircularPlanner`는 현재처럼 렌더링 전용 컴포넌트로 축소
4. 앱 전환 시에는 같은 상태 전이와 계약을 사용하는 React Native 전용 플래너 화면을 새로 구현

실행 기준:

- 앱에서 그대로 가져갈 것은 "렌더링 결과"가 아니라 "상태 전이와 규칙"이다.
- 웹 UI에서 추가 분리가 필요해지면 `use-planner-state` 아래에 앱 공용 view-model 계층을 두는 방향을 우선 검토한다.
- 하이브리드 래핑보다 React Native 재구성이 맞는 이유도 바로 이 계층 분리 기준에서 설명돼야 한다.

## Boundary Checklist Before App Transition

앱 전환 전에 아래 경계는 더 분리돼 있어야 한다.

1. 계획 저장소
현재 `src/providers/plans/plans-store.ts` 계약과 `src/providers/plans/local-plans.ts` 웹 구현이 분리돼 있다.
다음 단계는 앱 저장소 provider를 같은 계약으로 교체할 수 있게 유지하는 것이다.

2. 시간 소스 주입
현재 시간 소스는 `PlannerShell`을 통해 UI에 주입된다.
다음 단계는 이 주입 구조를 유지한 채 앱 시간 소스로 교체 가능하게 만드는 것이다.

3. 플래너 편집 상태와 도메인 규칙
현재 편집 상태와 저장/검증 호출은 `src/ui/planner/use-planner-state.ts`로 분리돼 있다.
다음 단계는 이 상태 전이 구조를 앱 UI에서도 재사용할 수 있게 유지하는 것이다.

4. 리마인드 provider
`src/providers/reminders/reminder-provider.ts`로 웹/앱 공통 알림 계약이 정리돼 있다.
현재는 `noop-reminder-provider`가 기본 구현이며, 다음 단계는 웹 프로토타입 또는 앱 네이티브 구현으로 교체하는 것이다.

## Boundary Split Order

앱 전환 전 분리는 아래 순서로 진행한다.

1. 계획 저장소 인터페이스 분리
- 현재 `localStorage` 직접 접근을 provider 계약 뒤로 숨기는 것이 가장 먼저다.
- 이유: 저장소 교체 비용이 낮고, 이후 편집 상태 분리와 앱 저장소 이행의 기반이 된다.

2. 시간 소스 주입 경로 분리
- 현재 시각 공급을 UI 밖에서 주입 가능하게 만든다.
- 이유: 앱 시간 소스 교체와 테스트 제어를 동시에 쉽게 만든다.

3. 편집 상태와 저장 호출 분리
- `circular-planner.tsx` 안의 폼 상태, 저장, 검증 호출을 한 단계 밖으로 뺀다.
- 이유: 웹 UI와 앱 UI가 다른 렌더링 레이어를 가져도 같은 편집 흐름을 재사용하기 쉬워진다.

4. 리마인드 provider 인터페이스 추가
- 최소 리마인드 경험을 웹/앱 공통 계약으로 먼저 정의한다.
- 이유: 앱 전환 직전이 아니라 그 전에 제품 요구 수준을 계약으로 고정해야 한다.
- 현재 상태: `ReminderProvider`는 `sync`, `schedule`, `cancel` 계약을 가지며, 웹 기본 구현은 `noopReminderProvider`다.

## Initial App Client Scope

초기 앱 클라이언트는 아래 범위를 최소 출시 기준으로 삼는다.

- 오늘 화면의 24시간 원형 플래너
- 계획 등록, 수정, 삭제
- 시작/종료 시간 자유 입력과 시간 겹침 차단
- 현재 시각 포인터와 현재 계획 강조
- 완료 토글과 하루 완료 현황 요약
- 로컬 저장소 기반 데이터 유지
- 최소 1종의 로컬 리마인드 또는 앱 내 리마인드

초기 앱 클라이언트에서 제외하는 항목:

- 가족 공유
- 외부 캘린더 연동
- 고급 통계/리포트
- 반복 일정 자동화
- 서버 기반 협업/동기화

이 범위는 웹 MVP에서 이미 검증 중인 핵심 흐름과 앱에서만 필요한 최소 리마인드 경험을 합친 것이다.

## What To Validate On Web First

앱 전환 전에 웹 MVP에서 더 검증해야 하는 항목:

- 원형 플래너가 목록형보다 현재 해야 할 계획 인지에 실제로 더 유리한지
- 시간 입력과 수정 흐름이 첫 사용자에게 부담스럽지 않은지
- 현재 계획 강조와 완료 처리 흐름이 한 세션에서 자연스럽게 이어지는지
- 최소 리마인드 경험이 실천 전환에 의미가 있는지

## UX Validation Priorities

앱 전환 우선안 확정 전에 남은 UX 검증은 아래 순서로 본다.

### Priority 1

- 원형 플래너가 첫 화면에서 "지금 해야 할 계획" 인지에 실제로 도움이 되는지
- 이유: 이 제품의 핵심 차별점이자 앱에서도 유지할지 말지를 결정하는 기준이기 때문이다.
- 확인 방식: 첫 화면 노출 후 사용자가 현재 해야 할 계획을 설명할 수 있는지 관찰

### Priority 2

- 시간 입력과 수정 흐름이 첫 사용자에게 과도한 부담을 주지 않는지
- 이유: 앱으로 옮겨도 시간 입력 UX가 유지 가능한지 판단해야 하기 때문이다.
- 확인 방식: 첫 계획 등록 완료 시간, 수정 성공 여부, 혼란 지점 기록

### Priority 3

- 현재 계획 강조에서 완료 처리까지의 흐름이 한 세션에서 자연스럽게 이어지는지
- 이유: 계획 인지와 실행 연결이라는 제품 목표를 직접 검증하는 흐름이기 때문이다.
- 확인 방식: 현재 계획 확인 후 완료 토글까지 이어지는 세션 존재 여부 확인

### Priority 4

- 최소 리마인드 경험이 실천 전환에 실제 의미가 있는지
- 이유: 앱 전환 후 네이티브 알림 구현 우선순위를 정하는 근거가 되기 때문이다.
- 확인 방식: 앱 내 리마인드 또는 로컬 리마인드 프로토타입 반응 관찰

## Current UX Evidence

현재 확보된 증거:

- 계획 등록, 수정, 삭제, 완료 토글의 기본 흐름은 Vitest와 Playwright로 고정돼 있다.
- 현재 계획 강조, 시간 겹침 차단, 요약 계산은 구현과 테스트로 유지되고 있다.

아직 부족한 증거:

- 실제 사용자가 첫 화면만 보고 현재 해야 할 계획을 바로 이해하는지에 대한 정성 확인
- 시간 입력 UX가 앱에서도 유지할 가치가 있는지에 대한 관찰 기록
- 리마인드가 실천 전환에 의미 있는지에 대한 검증 기록

## Reminder Provider Direction

현재 기준 결정은 "웹에서 얇은 리마인드 프로토타입을 먼저 붙이고, 네이티브 로컬 알림은 앱 전환 단계에서 구현"이다.

이 결정을 택하는 이유:

- 지금 부족한 것은 알림 API 구현 자체가 아니라, 리마인드가 실제로 실천 전환에 의미가 있는지에 대한 제품 증거다.
- 브라우저 Notification API를 먼저 붙여도 앱 전환 후 그대로 가져가기 어렵고, 권한/브라우저 제약 때문에 검증 노이즈가 커질 가능성이 높다.
- 이미 `ReminderProvider` 계약이 있으므로 웹에서는 앱 내 배너, 강조, 상태 메시지 같은 얇은 프로토타입을 같은 계약 뒤에 연결해 의미만 검증할 수 있다.
- 실제 로컬 푸시/백그라운드 알림은 `Expo + React Native` 전환 이후 구현하는 편이 최종 목표와 더 직접적으로 맞다.

권장 순서:

1. 웹에서는 `ReminderProvider`를 앱 내 리마인드 프로토타입에 연결
2. 이 프로토타입으로 리마인드가 완료 전환에 의미가 있는지 관찰
3. 의미가 확인되면 앱 전환 시 네이티브 로컬 알림 provider로 교체

웹 프로토타입에서 검증할 것:

- 사용자가 현재 계획 시작 시점에 리마인드를 알아차리는지
- 리마인드가 완료 토글까지 이어지는 흐름을 늘리는지
- 리마인드 빈도나 표현이 방해보다 도움이 되는지

웹 단계에서 보류할 것:

- 브라우저 Notification API 권한 UX 최적화
- 백그라운드 알림 보장
- 플랫폼별 로컬 알림 스케줄링 세부 구현

## Minimum Reminder Prototype On Web

웹 프로토타입은 아래 최소 경험만 제공한다.

포함할 것:

- 현재 시각이 계획 시작 시점에 가까워졌을 때 앱 내부에서 한 번 보이는 리마인드 신호
- 사용자가 리마인드에서 바로 현재 계획 제목과 시간을 확인할 수 있는 정보
- 리마인드 노출 뒤 완료 토글로 바로 이어질 수 있는 짧은 경로
- 같은 계획에 대해 중복으로 계속 쌓이지 않는 단일 리마인드 상태

포함하지 않을 것:

- 브라우저 바깥 시스템 알림
- 여러 시점에 반복되는 다단계 리마인드
- 스누즈, 미루기, 반복 주기 설정
- 권한 요청 플로우
- 백그라운드 지속 보장

초기 표시 원칙:

- 첫 단계에서는 "계획 시작 직전 또는 시작 시점 근처 1회"만 본다.
- 리마인드 표현은 배너, 인라인 강조, 상태 메시지 중 하나로 충분하다.
- 제품 검증 목적은 알림 기술이 아니라 "알아차림 -> 완료 전환"이 생기는지 확인하는 것이다.

성공 기준:

- 사용자가 리마인드를 보고 현재 해야 할 계획을 바로 인지할 수 있다.
- 리마인드 이후 완료 토글까지 이어지는 흐름이 실제로 관찰된다.
- 리마인드가 방해로 느껴진다는 피드백보다 도움이 된다는 반응이 더 많다.

## What Can Be Decided Now

웹 추가 검증 없이 지금 바로 결정 가능한 항목:

- 도메인 로직, 시간 계산, 저장소, 알림, 시간 소스는 앱 전환 전 분리 유지
- 초기 앱 클라이언트는 단일 사용자 기준으로 시작
- 가족 공유, 고급 통계, 외부 캘린더 연동은 초기 앱 범위에서 제외
- 웹 MVP는 최종 제품이 아니라 앱 전환 전 검증 단계로 취급
- 리마인드는 웹에서 얇은 프로토타입으로 의미만 검증하고, 네이티브 로컬 알림은 앱 전환 단계에서 구현

## Option Assessment

현재 문서와 코드 기준의 중간 판단:

### Expo + React Native

- 장점: 최종 목표가 실제 앱인 만큼, 로컬 알림과 모바일 상호작용에 가장 직접적으로 맞는다.
- 장점: 현재 UI는 웹 전용 결합이 강해 어차피 재구성이 필요하므로, 재구성 비용이 하이브리드 대비 압도적으로 불리하지만은 않다.
- 장점: 도메인 로직과 규칙을 그대로 옮기고, UI만 앱 기준으로 다시 설계하기 좋다.
- 단점: 원형 플래너와 입력 흐름을 React Native 기준으로 새로 구현해야 한다.

### Hybrid Wrapper

- 장점: 웹 렌더링을 빠르게 앱 셸에 넣을 수 있다.
- 장점: 초기 화면 재사용 속도는 빠를 수 있다.
- 단점: 최종 목표가 앱 품질과 로컬 알림 경험이라면 중장기 제약이 커질 가능성이 높다.
- 단점: 현재 웹 UI가 브라우저 API와 폼 상호작용에 묶여 있어, 단순 래핑만으로 제품 목표를 충분히 달성하기 어렵다.

## Current Recommendation

- 현재 기준 권장안은 `Expo + React Native`다.
- 단, 이 결정을 확정하기 전 아래 두 항목은 더 확인한다.
- 웹 MVP에서 원형 플래너와 시간 입력 UX가 실제로 유지할 가치가 있는지
- 앱 전환 전에 저장소/알림 provider를 얼마나 분리해야 하는지
- 1단계: 모바일 우선 웹 MVP 구축
- 2단계: 사용자 검증 및 핵심 흐름 측정
- 3단계: 전환 조건 충족 시 `Expo + React Native` 우선안 기준으로 앱 전환 실행 준비
- 4단계: 추가 검증에서 치명적 반례가 없으면 앱 전환 방식을 확정

## Required Architecture Rules

- 도메인 로직은 렌더링 코드와 분리한다.
- 저장소 접근은 인터페이스 뒤에서만 수행한다.
- 알림 기능은 provider로 캡슐화한다.
- 현재 시각 계산은 테스트 가능한 순수 함수로 유지한다.
