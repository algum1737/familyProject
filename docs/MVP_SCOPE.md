# MVP Scope

## Goal

`오늘 다 했니?`의 첫 버전은 사용자가 하루 계획을 원형 24시간 플래너에 배치하고, 현재 시각 기준으로 지금 해야 할 계획을 직관적으로 확인하고, 완료 여부를 기록하는 가장 작은 작동 단위를 제공하는 데 목적이 있다.

## Product Decision

- 대상 사용자: 개인 사용자 1인
- 우선 플랫폼: 모바일 우선 앱 스타일 웹앱
- 범위 기준 시간 단위: 하루
- 제품 핵심 가치: 계획을 실제 실행으로 연결하고, 하루 완료 상태를 빠르게 확인하게 한다
- 핵심 인터페이스: 원형 24시간 계획판과 현재 시각을 가리키는 시침형 표시

## In Scope

- 원형 24시간 계획판 기반 오늘 화면
- 오늘의 계획 항목 등록
- 계획 항목별 시작 시간과 종료 시간 입력
- 일정 시간은 분 단위로 자유롭게 입력 가능
- 현재 시각에 따라 회전 또는 갱신되는 시침형 현재 시간 표시
- 현재 시각이 포함된 계획 구간의 시각적 강조
- 오늘 화면에서 계획 목록 확인
- 항목 완료/미완료 상태 변경
- 오늘 완료율 또는 완료 개수 요약 표시
- 예정 시간 기준 로컬 리마인드 또는 앱 내 리마인드 구조 준비

## Out Of Scope

- 가족 공유
- 다중 사용자 협업
- 소셜 기능
- 고급 통계/리포트
- 반복 일정 자동화
- 외부 캘린더 연동
- 서버 기반 푸시 알림
- 복잡한 권한/조직 기능

## MVP Core User Journey

1. 사용자가 오늘 할 일을 1개 이상 등록한다.
2. 각 할 일에 시작 시간과 종료 시간을 자유롭게 설정한다.
3. 사용자는 원형 24시간 계획판에서 하루 일정을 한눈에 본다.
4. 시스템은 현재 시각을 시침형 포인터로 표시한다.
5. 사용자는 지금 시점에 해야 할 계획이 무엇인지 즉시 확인한다.
6. 적절한 시점에 리마인드를 확인한다.
7. 각 항목을 완료 처리한다.
8. 하루 종료 전에 완료 현황을 확인한다.

## MVP Screens

- 오늘 화면: 원형 24시간 계획판 + 현재 시각 포인터 + 현재 해야 할 계획 강조
- 계획 추가/수정 화면
- 완료 현황 요약 영역

## Functional Requirements

- 사용자는 하루 계획 항목을 생성할 수 있어야 한다.
- 각 계획 항목은 제목, 시작 시간, 종료 시간을 가질 수 있어야 한다.
- 시간 입력은 분 단위로 자유롭게 가능해야 한다.
- 이미 등록된 시간과 겹치는 일정은 저장할 수 없어야 한다.
- 사용자는 항목 완료 여부를 즉시 변경할 수 있어야 한다.
- 시스템은 오늘 기준 계획만 명확하게 보여줘야 한다.
- 시스템은 24시간 원형 레이아웃에 계획 구간을 배치해 보여줘야 한다.
- 시스템은 현재 시각을 실시간 또는 분 단위 갱신으로 표시해야 한다.
- 시스템은 현재 시각이 속한 계획 구간을 다른 구간보다 직관적으로 강조해야 한다.
- 시스템은 완료 개수 또는 완료율을 계산해 보여줘야 한다.

## Non-Functional Requirements

- 첫 사용자가 3분 안에 첫 계획을 등록할 수 있어야 한다.
- 주요 흐름은 모바일 화면 폭에서도 무리 없이 동작해야 한다.
- 상태 변경은 즉시 UI에 반영되어야 한다.
- 원형 계획판은 현재 계획과 비현재 계획이 시각적으로 명확히 구분되어야 한다.
- 현재 시간 포인터는 사용자가 별도 계산 없이 지금 해야 할 일을 이해하게 해야 한다.
- 핵심 흐름은 에이전트가 DOM 또는 테스트로 검증 가능해야 한다.

## Success Criteria

- 사용자가 하루 계획 3개를 등록할 수 있다.
- 사용자가 각 계획의 시작/종료 시간을 자유롭게 설정할 수 있다.
- 사용자가 등록한 3개 항목 중 1개 이상을 완료 처리할 수 있다.
- 사용자가 현재 시각 기준으로 지금 해야 할 계획을 한눈에 식별할 수 있다.
- 사용자가 오늘 완료 현황을 한 화면에서 확인할 수 있다.
- 핵심 사용자 여정에 대한 smoke test를 만들 수 있다.

## Transition Gate

웹 MVP 검증이 아래 조건을 만족하면 앱 전환을 시작할 수 있다.

- 핵심 사용자 여정 smoke test가 안정적으로 통과한다.
- 사용자 테스트 또는 운영 데이터에서 "지금 해야 할 계획을 이해할 수 있다"는 신호가 반복 확인된다.
- 일정 등록 -> 현재 계획 인지 -> 완료 처리 흐름이 한 세션에서 자연스럽게 이어진다.
- 웹 MVP에서 치명적인 UX 혼란이 해결됐다.
- 앱 전환 시 재사용할 도메인 로직과 provider 경계가 분리되어 있다.

## Open Decisions

- 기술 스택: React Native, Expo, PWA, 또는 모바일 우선 웹앱 중 무엇을 택할지
- 리마인드 방식: 브라우저 알림, 앱 내 리마인드, 로컬 알림 중 무엇을 택할지
- 저장 방식: 로컬 저장소 우선인지 서버 동기화까지 포함할지
- 계정 전략: 익명 시작 후 확장할지, 처음부터 로그인할지
- 원형 계획판 편집 UX를 드래그 기반으로 할지 입력 폼 기반으로 시작할지
- 앱 전환 방식: React Native 재구현인지, 하이브리드 래핑인지

## Recommended Implementation Order

1. 원형 24시간 계획판 UI 골격 구축
2. 오늘 계획 CRUD
3. 시작/종료 시간 자유 입력
4. 현재 시각 포인터와 현재 계획 강조
5. 완료 상태 토글과 요약 계산
6. 로컬 저장
7. 리마인드 구조 추가
8. 핵심 여정 테스트 추가
