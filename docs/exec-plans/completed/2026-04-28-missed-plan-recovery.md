# Missed Plan Recovery Plan

## Goal

종료 시각이 지난 미완료 일정을 `missed` 상태로 전환하고, 이후 사용자가 회고 또는 같은 날 빈 시간 재지정을 선택할 수 있는 기반을 만든다.

## Why Now

- 현재는 일정이 끝나도 `pending`으로 남아 상태 의미가 약하다.
- 사용자는 놓친 일정을 완료와 구분해서 보고, 회고하거나 다시 잡을 수 있어야 한다.
- 앱 전환 시 종료 5분 전 알림 정책도 이 상태 모델 위에서만 자연스럽게 설계된다.

## Scope

- `pending`, `done` 외에 `missed` 상태 추가
- 종료 시각 경과 시 `pending -> missed` 자동 전환
- 재지정 최대 3회를 위한 메타데이터 추가
- 회고/재지정 흐름의 웹 MVP 규칙 정의
- 상태 라벨 커스터마이징 요구를 데이터 모델 후보로 정리
- 앱 전환 문서에 종료 5분 전 회고/재지정 알림 방향 반영

## Non-Goals

- 이번 단계에서 회고 입력 UI 전체 구현
- 이번 단계에서 재지정 UX 전체 구현
- 이번 단계에서 사용자 설정 화면 구현
- 이번 단계에서 네이티브 알림 구현

## State Model Draft

- `pending`: 시작 전 또는 진행 중이며 아직 완료하지 않은 일정
- `done`: 사용자가 완료 처리한 일정
- `missed`: 종료 시각이 지났지만 완료하지 못한 일정

`지금`은 저장 상태가 아니라 현재 시각으로 계산되는 파생 표시값으로 유지한다.

## Recovery Draft

- `missed` 일정은 사용자가 `회고` 또는 `다시 지정` 중 하나를 선택할 수 있어야 한다.
- `다시 지정`은 같은 날 빈 시간에만 가능하다.
- 재지정 횟수는 최대 3회다.
- 원래 일정 길이를 채울 연속 빈 시간이 없으면 자동 축약이나 자동 분할 없이 다시 지정 실패로 처리한다.
- 재지정 일정도 다시 놓치면 다시 `missed`가 될 수 있다.

## Label Customization Draft

- 상태 값은 고정하고, `놓침`, `완료`, `지금` 같은 표시 문구만 사용자 설정으로 바꾼다.
- 라벨 커스터마이징은 저장 상태 모델과 분리된 별도 설정 계층으로 둔다.
- 앱 공통 안정 범위는 `current`, `pending`, `done`, `missed`, `completeNow`, `reflection`, `reschedule` 7개 키로 제한한다.
- 섹션 제목, 안내 문구, 검증 오류 메시지는 초기 앱 범위에서는 제품 고정 문구로 둔다.

## Completion Status

Completed. `missed` state, reflection, rescheduling, recovery highlighting, chain-based completion accounting, and next-day rescheduled follow-up cleanup are implemented and tested.

## Open Work

- None for this execution plan.
- Future notification or copy policy changes should start as new active plans.

## Recent Changes

- `missed` 상태와 `rescheduleCount` 메타데이터를 도메인 모델에 반영
- 종료 시각 경과 시 `pending -> missed` 자동 전환 규칙을 웹 MVP에 연결
- `missed` 일정의 `회고` 메모 저장 UI를 추가
- `missed` 일정의 `다시 지정` 흐름을 기존 계획 폼과 연결하고, 같은 날 빈 시간 새 일정 생성과 최대 3회 제한을 구현
- 회고가 저장된 일정은 리스트에서 `회고 저장됨`과 메모 미리보기로 다시 보이게 했고, 재지정 이력은 `다시 지정 n/3`로 남긴다
- 상태 버튼과 리마인드/놓침 액션 라벨은 별도 설정 계층과 `localStorage` 저장 구조 뒤로 분리했고, 웹 UI에서 바로 수정/복원할 수 있게 했다
- 라벨 설정 UI는 인라인 편집 대신 `표시 문구 변경` 팝업으로 정리했고, 각 항목은 최대 10자까지만 입력 및 저장되게 제한했다
- 앱 전환 문서에는 종료 5분 전 회고/재지정 알림의 1회 노출과 기본 액션 정책을 추가했다
- 회고가 없는 `missed` 일정은 `회고 다시 보기`, 멀리 떨어진 후속 일정은 `다시 지정됨`, 곧 시작하는 후속 일정은 `다시 지정 곧 시작`, 현재 진행 중인 후속 일정은 `회복 진행 중`으로 다시 강조한다
- 초기 앱 범위에서는 라벨 커스터마이징을 상태 4개와 핵심 액션 3개로 고정하고, 설명성 문구는 제품 고정 문구로 유지하기로 했다
- `회복 관찰 로그`는 재강조 빈도와 함께 `관찰 더 필요`, `조정 검토`, `우선 유지` 판단도 보여준다
- 연속 빈 시간이 부족해 다시 지정이 막히면 `다시 지정 불가` 관찰 로그를 남겨 재지정 정책의 엄격도를 나중에 판단할 수 있게 했다
- `docs/WEB_TO_APP_TRANSITION.md`에 회복 재강조 관찰 런북과 종료 5분 전 알림 검토 체크리스트를 추가해, 다음 단계가 실제로 무엇을 기록하고 어떤 기준으로 판단할지 고정했다
- `docs/RECOVERY_OBSERVATION_LOGBOOK.md`를 추가해 관찰 세트와 주간 요약을 같은 형식으로 남길 수 있게 했다
- `docs/RECOVERY_OBSERVATION_LOGBOOK.md`에 Safari 비공개 브라우징 기준 초기 3세트 탐색 기록을 남겨, 회고 저장 흐름과 다시 지정 불가 반응을 실제로 문서화했다
- 개발 모드 전용 `관찰 시간 조정` 패널을 추가해 시작 5분 전, 시작 직후, 종료 직전 표본을 반복 재현할 수 있게 했다
- 같은 패널로 `저녁` 시작 직후 완료 1회와 `운동` 시작 5분 전 닫기 1회를 실제로 기록해 첫 리마인드 `표시/닫기/완료` 표본을 확보했다
- 같은 패널에 `표본용 예정 일정 추가` 버튼을 더해 미래 30분 `pending` 표본을 즉시 만들 수 있게 했고, `관찰 표본 1`, `관찰 표본 2`, `관찰 표본 3`까지 포함해 누적 `표시 5회 / 닫기 2회 / 완료 2회`에서도 정책 상태 `우선 유지`가 유지되는 것을 다시 확인했다
- 같은 `관찰 표본 2` 체인에서 `시작 5분 전 닫기 -> 종료 직후 missed -> 회고 다시 보기`, `관찰 표본 3` 체인에서 `시작 리마인드 표시 -> 종료 5분 전 추가 알림 없이 유지 -> 종료 직후 missed -> 회고 다시 보기` 흐름이 자연스럽게 이어지는 것도 추가 확인했다
- 개발 모드에만 노출되는 `종료 전 확인` 실험 배너를 추가했고, 액션은 `계속 진행` 1개로 제한했다
- 실제 관찰에서 `관찰 표본 1` 체인 기준 `종료 전 확인 -> 계속 진행 -> 종료 직후 missed -> 회고 다시 보기` 흐름이 자연스럽게 이어지는 것도 추가 확인했다
- 같은 날 `관찰 표본 2` 체인에서도 `시작 리마인드 -> 종료 전 확인 -> 계속 진행 -> 종료 직후 missed -> 회고 다시 보기`가 다시 반복되는 것을 확인했다
- 시작 리마인드 누적은 `표시 7회 / 닫기 2회 / 완료 2회`까지 반영했다
- 종료 5분 전 알림 최종 판단을 `docs/END_5_MINUTE_ALERT_DECISION.md`와 `docs/WEB_TO_APP_TRANSITION.md`에 문서화했다
- 회복 관찰 로그에도 `종료 전 확인`, `계속 진행` 이벤트를 남겨 실제 운영 추적 기반을 추가했다
- `다시 지정 불가` 오류는 `길이 그대로는 불가`와 `더 짧은 새 시간으로 다시 지정 저장` 안내로 나눠 후속 행동을 더 직접적으로 보여주게 정리했다

## Validation

- 기존 `pending/done` 테스트가 깨지지 않아야 한다.
- 종료 시각이 지난 일정은 자동으로 `missed`가 되어야 한다.
- `missed` 상태는 `done`과 구분돼 렌더링되어야 한다.
- 재지정 메타데이터는 기존 저장 데이터와 호환돼야 한다.
