# Plan Editing UI Plan

## Goal

웹 MVP의 계획 관리 흐름을 실제 사용 가능한 수준으로 올린다.

## Scope

- 계획 수정 모드 추가
- 수정 모드에서 등록 폼 재사용
- 시간 겹침 검증 유지
- 색상 선택 UX 단순화
- 수정/등록 모드 버튼 레이아웃 분기

## Current State

- 이 계획의 주요 구현 결과는 현재 `main` 기준선에 반영돼 있다.
- 계획 등록, 삭제, 완료 토글은 이미 동작한다.
- 수정 버튼을 누르면 폼이 기존 값으로 채워지고 수정 모드로 전환된다.
- 수정 모드에서는 섹션 제목이 `계획 수정`, 제출 버튼이 `계획 저장`으로 바뀐다.
- 색상 선택은 기본 팔레트 드롭다운 + `사용자 지정` 컬러 피커로 정리됐다.
- 시간 입력은 자유 분 단위이며, 기존 일정과 겹치는 시간은 저장되지 않는다.
- 충돌 메시지는 새로 등록하려는 일정이 아니라 기존 등록 일정의 제목과 시간대로 안내된다.
- 여러 기존 일정과 겹치더라도 첫 번째로 감지된 충돌 일정 1건만 안내된다.

## Recent Changes

- 수정 모드 취소 버튼 추가
- 수정 모드 버튼 폭 축소
- 등록 모드 `계획 추가` 버튼 최소 너비를 `180px`로 유지
- 수정 모드 액션 영역이 왼쪽 입력 필드를 침범하지 않도록 정렬 조정
- 색상 미리보기는 폼 하단 별도 블록으로 유지
- `시간 자유 입력` 보조 문구 제거
- 액션 버튼은 입력 필드들과 같은 라인에 정렬
- 충돌 메시지를 기존 등록 일정 제목/시간 기준으로 구체화
- 첫 충돌 일정 1건만 안내하도록 정책 정리
- Vitest UI 사용자 흐름 테스트와 Playwright 브라우저 E2E 흐름 테스트 추가
- GitHub Actions CI에서 문서 검증, 타입체크, Vitest, Playwright E2E를 자동 실행하도록 구성
- Playwright는 로컬에서는 시스템 Chrome 채널, CI에서는 번들 `chromium`을 사용하도록 분기
- `scripts/check-handoff-loop.sh`로 `HANDOFF` 갱신과 완료 계획 이동 점검을 한 번에 확인하도록 묶음
- `.githooks/post-commit`에서 커밋 직후 해당 점검 스크립트를 실행하도록 연결
- 점검 스크립트가 `src/`, `tests/`, `package.json`, CI 워크플로, 실행 계획 위치 변경까지 함께 경고하도록 보강
- `npm run check:handoff-loop`로 훅 밖에서도 같은 점검을 수동 실행할 수 있게 함
- `npm run test:handoff-loop`로 대표 커밋 시나리오에서 경고 품질을 자동 검증할 수 있게 함

## Fixed Test Baseline

계획 편집 흐름은 아래 범위를 기본 검증 세트로 고정한다.

### Domain Unit Tests

- `validatePlanner`는 겹치는 일정 저장을 막아야 한다.
- 겹침 오류 메시지는 새 일정이 아니라 기존 등록 일정의 제목과 시간대로 안내해야 한다.
- 인접한 일정은 겹침 없이 허용해야 한다.
- 현재 시각 기준 현재 계획 판정과 완료 요약 계산은 유지돼야 한다.

### Vitest UI Flows

- 새 계획 추가 시 정렬 순서와 `localStorage` 저장 결과가 함께 유지돼야 한다.
- 완료된 계획을 수정해도 완료 상태가 보존돼야 한다.
- 겹침 저장 시 기존 일정 기준 충돌 메시지가 노출되고 새 일정은 추가되지 않아야 한다.
- 수정 중인 계획을 삭제하면 폼이 등록 모드 초기 상태로 돌아가야 한다.

### Playwright Browser Flows

- 브라우저에서 계획 추가 후 겹침 저장 차단까지 한 흐름으로 검증해야 한다.
- 브라우저에서 계획 수정, 완료 토글, 삭제까지 한 흐름으로 검증해야 한다.

## Deferred Coverage

아래 항목은 현재 기본 고정 범위에는 넣지 않고 확장 후보로 남긴다.

- 기본 팔레트와 `사용자 지정` 색상 선택 세부 상호작용
- 수정 모드 `취소` 버튼 흐름
- 여러 일정과 동시에 겹칠 때 첫 충돌 일정 1건만 노출되는지에 대한 UI/E2E 검증
- 새로고침 이후 저장 데이터 재수화 확인
- 현재 계획 강조와 시계 라벨의 시각 회귀 검증

## Deferred Coverage Trigger

아래 조건 중 하나가 생기면 확장 후보를 기본 검증 세트 편입 대상으로 다시 검토한다.

- 계획 편집 UI를 다시 바꾸면서 기존 테스트가 잡지 못한 회귀가 발생했을 때
- 색상 선택, 수정 취소, 재수화 같은 보조 흐름에서 실제 버그가 한 번이라도 확인됐을 때
- 새로운 사용자 흐름이 MVP 핵심 여정에 편입돼 릴리스 차단 조건이 됐을 때
- CI는 통과하지만 수동 점검에서 반복적으로 같은 회귀가 발견될 때

우선순위는 아래 순서를 따른다.

1. 실제 회귀가 발생한 흐름
2. 저장/복구처럼 데이터 손실 위험이 있는 흐름
3. 사용 빈도가 높은 편집 보조 흐름
4. 시각 회귀처럼 기능 차단성이 낮은 흐름

## CI Failure Response

CI 실패 시 후속 조치는 아래 순서를 기본으로 한다.

1. 실패한 명령이 `validate:docs`, `typecheck`, `test`, `test:e2e` 중 무엇인지 확인한다.
2. 실패 원인이 문서 불일치나 계획 문서와 구현 상태의 불일치라면 같은 작업에서 `HANDOFF`와 관련 실행 계획을 함께 갱신한다.
3. 실패 원인이 구현 버그인데 문서 상태 설명도 더 이상 맞지 않게 됐다면, 코드 수정과 함께 문서도 갱신한다.
4. 실패 원인이 일시적 환경 문제라면 재현 조건과 로그만 남기고 `HANDOFF`의 현재 상태 서술은 바꾸지 않는다.

문서 갱신이 필요한 실패의 예시는 아래와 같다.

- 테스트 범위가 바뀌어 실행 계획의 `Fixed Test Baseline`이 더 이상 맞지 않을 때
- 브라우저 실행 조건이 바뀌어 `HANDOFF`의 E2E 실행 기준 설명이 틀렸을 때
- 완료된 계획이 다시 `active/`에 남거나, 반대로 미완료 계획이 `completed/`로 이동된 것이 확인됐을 때
